## 完成的开发内容

1. **新增 `OnlineVerificationOperator` 算子**
   这个核心引擎模块内置了完整的“一条龙”流水线：
   - 自动挂载上下文的 `qr_codes`
   - 使用配置的命名正则表达式提取二维码内的 `ID` 和 `校验码` 等参数。
   - 自动使用提取出的变量替换 `URL` 模板。
   - 智能调用原本的 `DocumentDiffOperator` 发起远程下载、解析以及大文本极速差异比对，并进行 Github 风格的高亮标识。

## 数据库迁移说明 (针对已有数据库)

如果你的系统是**全新部署**的，SQLAlchemy 会自动初始化包含这个新枚举值的表结构，无需任何操作。

但是，对于**已存在的数据库环境**（例如你当前的测试环境），由于 PostgreSQL 对 `ENUM` 枚举类型的校验极其严格，后端代码中新增了 `online_verification` 类型后，底层数据库并不会自动添加该枚举，导致前端在保存模块时报错。

为了方便后续迁移，我已经在项目根目录创建了 `db_migrations.sql` 脚本。如果需要在其他已有环境部署此功能，请在数据库内直接执行以下命令更新枚举：
```sql
ALTER TYPE moduletype ADD VALUE IF NOT EXISTS 'online_verification';
```

## 你现在的体验与使用步骤

从现在开始，操作变得非常清爽：

1. 打开 **“校验模块管理”** 页面，点击右上角的 **“新建模块”**。
2. 在下拉框选择：**“🔗 在线防伪比对 (一体化)”**。
3. 界面上会自动多出三个让你填写的配置项：
   - **二维码提取正则**：（默认已给你填入了示范 `id:(?P<report_id>\d+);(?P<verify_code>\w+)`，你可以随意改成你的业务正则）
   - **请求URL模板**：（示范：`https://api.example.com/check?id={{report_id}}&code={{verify_code}}`）
   - **相似度报警阈值(%)**：（默认 `95`）
4. 保存模块（例如取名叫 “SGS 防伪拉取”、“TUV 原件比对” 等）。
5. 回到“规则配置中心”，添加或编辑规则时，直接在 **“关联校验模块”** 里把它们打上勾 ✅ 即可。
